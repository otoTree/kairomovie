# 记忆系统设计与优化

本文档深入设计了 Kairo 在云端环境下的记忆管理系统，重点解决有限上下文窗口与大规模信息交互之间的矛盾。

## 1. 记忆分层架构

我们将记忆分为三层，对应不同的访问频率和存储介质。

### 1.1 工作记忆 (Working Memory)
-   **定义**: 当前正在进行的任务上下文。
-   **存储**: Redis / 内存 (Agent 运行时)。
-   **内容**:
    -   当前的 System Prompt。
    -   最近 N 轮对话 (滑动窗口，例如最近 10 轮)。
    -   当前任务的目标 (Task Goal)。
    -   当前步骤的中间状态 (Scratchpad)。
-   **特点**: 读写极快，随任务结束而清除或归档。

### 1.2 短期记忆 (Short-term Memory / Episodic)
-   **定义**: 项目内的历史交互记录。
-   **存储**: PostgreSQL (`events` 表)。
-   **内容**:
    -   过去任务的执行日志。
    -   用户对特定生成结果的反馈。
-   **检索方式**: 基于时间或任务 ID 查询。

### 1.3 长期记忆 (Long-term Memory / Semantic)
-   **定义**: 跨项目的知识、用户偏好、风格指南。
-   **存储**: PostgreSQL (`agent_memory` 表 + `pgvector`)。
-   **内容**:
    -   用户画像 ("喜欢快节奏剪辑")。
    -   项目背景资料 ("这是一个关于环保的纪录片系列")。
    -   从大文件中提取的关键事实。
-   **检索方式**: 向量相似度搜索 (Embedding Search)。

## 2. 记忆读取流程 (Retrieval)

为了在有限的上下文窗口中提供最相关的信息，我们采用 **"检索增强生成 (RAG)"** 模式。

### 2.1 上下文组装 (Context Construction)

每次向 LLM 发送请求前，动态构建 Prompt：

```text
[System Message]
你是一个专业的视频创作助手 Kairo。
当前任务: {current_task}

[Relevant Memories] (动态插入)
- 用户偏好: 视频风格应为 "赛博朋克"。
- 项目背景: 主角叫 "Neo"。
- 文件摘要: 上传的 PDF 文档提到 "2077年"。

[Conversation History] (最近 10 轮)
User: ...
Assistant: ...

[User Input]
{user_input}
```

### 2.2 动态检索策略
1.  **意图识别**: 分析用户输入的关键词（例如 "背景音乐"）。
2.  **向量搜索**: 在 `agent_memory` 中搜索与 "背景音乐" 相关的 Top-3 记录。
3.  **相关性过滤**: 过滤掉相关度低于阈值 (例如 0.7) 的记忆，避免噪声。
4.  **Token 预算控制**: 严格限制插入的记忆 Token 数量 (例如最多 500 Tokens)。

## 3. 记忆写入与更新 (Consolidation)

记忆写入是一个高频操作，需要优化以避免阻塞用户交互。

### 3.1 异步记忆提取 (Asynchronous Extraction)
不要在每轮对话后立即更新长期记忆。而是采用 **"后处理"** 机制：
1.  **对话结束/闲置时**: 触发 `MemoryExtractionJob`。
2.  **提取模型**: 使用轻量级模型 (如 GPT-3.5-Turbo 或本地小模型) 分析最近的对话。
3.  **提取指令**:
    > "分析以下对话，提取用户的新偏好、项目的新事实。忽略闲聊。输出 JSON 格式。"

### 3.2 记忆去重与融合
1.  **查询现有记忆**: 在插入新记忆前，先用新记忆的向量查询数据库。
2.  **相似度检查**: 如果存在相似度 > 0.9 的记录，则认为是重复或更新。
3.  **融合策略**:
    -   **更新**: 如果新记忆更具体，更新旧记录。
    -   **忽略**: 如果信息已存在，跳过。
    -   **新增**: 如果是全新信息，插入新记录。

## 4. 大文件与长文本处理

针对用户上传的剧本、小说等长文本，不能直接放入 Context。

### 4.1 预处理流程
1.  **分块 (Chunking)**: 将大文件按语义（段落、章节）切分为小块 (例如 500 Tokens/块)。
2.  **向量化 (Embedding)**: 计算每个块的 Embedding 并存入 `project_knowledge` 表。
3.  **摘要 (Summarization)**: 生成全文摘要，存入 `agent_memory` 作为高层索引。

### 4.2 读取策略
-   **概览**: 默认只加载全文摘要。
-   **按需检索**: 当用户问 "主角在第三章做了什么？" 时，Agent 触发检索工具，搜索相关文档块，将其临时加入 Context。

## 5. 性能优化

-   **缓存 (Caching)**: 对高频访问的记忆（如用户画像）使用 Redis 缓存。
-   **批量处理**: 累积多条记忆更新操作，批量写入数据库。
-   **向量索引**: 使用 `HNSW` 索引加速 `pgvector` 查询。
